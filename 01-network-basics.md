## 1. HTTP 协议深入

### 1.1 HTTP 是什么

**HTTP (HyperText Transfer Protocol)** 是 Web 的基石——浏览器和服务器之间传输数据的协议。

一句话定义：HTTP 是一种**无状态**的**应用层**协议，基于**请求-响应**模型，客户端发起请求，服务端返回响应。

> **无状态**：服务器不会记住"上一个请求是谁发的"，每次请求都是独立的。这就是为什么需要 Cookie/Session/Token 来维持登录状态。
>
> **生活化类比**：HTTP 的无状态就像**快餐店柜台**——收银员不会记住你上一顿点了什么。你每次来都得重新报一遍："我要一个巨无霸套餐"。Cookie/Session/Token 就像**会员卡**——你掏出卡一刷，收银员就知道你是谁、有什么偏好、上次积分多少。

### 1.2 HTTP 请求与响应结构

#### 请求报文结构

```
POST /api/users HTTP/1.1              ← 请求行：方法 + 路径 + 协议版本
Host: example.com                      ←
Content-Type: application/json         ← 请求头 (Headers)
Authorization: Bearer eyJhbGci...      ←
                                       ← 空行分隔
{                                      ←
  "name": "sam",                       ← 请求体 (Body)
  "email": "sam@example.com"          ←
}                                      ←
```

响应的结构完全对称，把**请求行**换成**状态行**（协议版本 + 状态码 + 状态描述）。

在 Chrome DevTools → Network → 点一个请求 → Headers 那一栏展示的就是完整报文。

### 1.3 HTTP 方法

| 方法 | 作用 | 幂等性 | 安全性 | 请求体 |
|------|------|--------|--------|--------|
| **GET** | 获取资源 | ✅ 是 | ✅ 是 | ❌ 无 |
| **POST** | 创建资源 | ❌ 否 | ❌ 否 | ✅ 有 |
| **PUT** | 完整替换资源 | ✅ 是 | ❌ 否 | ✅ 有 |
| **PATCH** | 部分更新资源 | ❌ 否 | ❌ 否 | ✅ 有 |
| **DELETE** | 删除资源 | ✅ 是 | ❌ 否 | ❌ 通常无 |
| **HEAD** | 只获取响应头（不要 Body） | ✅ 是 | ✅ 是 | ❌ 无 |
| **OPTIONS** | 查询支持的方法（预检请求） | ✅ 是 | ✅ 是 | ❌ 无 |

**两个关键概念：**

- **幂等性**：同一个请求发一次和发一百次，服务器的最终状态一样。`PUT`（"把用户年龄设为 25"）是幂等的；`POST`（"创建一个新用户"）不是——发一百次创建一百条。
- **安全性**：不修改服务器数据的叫安全方法（只读操作）。`GET`、`HEAD`、`OPTIONS` 是安全的。

> **生活化类比**：幂等性就像**电梯按钮按 3 楼**——你按一次还是按十次，电梯最终都停在 3 楼。而 POST 就像**点外卖下单**——你不小心点了三次"提交订单"，就会送来三份。安全性就是**看菜单 vs 下单**——看菜单（GET）不会让厨师开始炒菜，下单（POST）就会。

**实际项目里怎么用：**

```
GET    /api/users          → 获取用户列表
GET    /api/users/123      → 获取 id=123 的用户
POST   /api/users          → 创建新用户（数据放 Body）
PUT    /api/users/123      → 完整替换 id=123 的用户
PATCH  /api/users/123      → 部分更新（如只改邮箱）
DELETE /api/users/123      → 删除 id=123 的用户
```

### 1.4 HTTP 状态码

不需要背，理解分类即可，碰到不认识的去查。按百位数分类：

| 范围 | 含义 | 常见实例 |
|------|------|---------|
| **1xx** | 信息提示 | `101 Switching Protocols`（WebSocket 升级用） |
| **2xx** | 成功 | `200 OK`、`201 Created`（POST 创建成功）、`204 No Content`（DELETE 成功但没东西返回） |
| **3xx** | 重定向 | `301 Moved Permanently`（永久迁移，SEO 权重转移）、`302 Found`（临时跳转）、`304 Not Modified`（缓存没变，用本地缓存） |
| **4xx** | 客户端错误 | `400 Bad Request`（请求格式有问题）、`401 Unauthorized`（没登录）、`403 Forbidden`（登录了但没权限）、`404 Not Found`、`429 Too Many Requests`（限流） |
| **5xx** | 服务端错误 | `500 Internal Server Error`、`502 Bad Gateway`（上游挂了）、`503 Service Unavailable` |

**前端必须敏感的几个状态码：**

- `401` → 引导用户重新登录
- `403` → 提示"你没有权限"
- `304` → 你知道命中了缓存，不用额外处理
- `502`/`503` → 后端挂了，提示用户稍后重试，不要重复提交

> **生活化类比**：状态码就像**快递包裹的物流状态**——`200` 是"已签收"，`301/302` 是"该地址已变更，请前往新地址"，`401` 是"请先登录/实名认证"，`403` 是"你没有权限查看这个包裹"，`404` 是"查无此单号"，`500` 是"快递员把你包裹弄丢了"。

### 1.5 HTTP Headers 详解

Headers 是 HTTP 的"元数据层"。你可以在请求或响应里附加键值对，控制缓存、认证、跨域、压缩等各种行为。

#### 通用类

> **生活化类比**：Headers 就是**信封上的标签**——信封上贴个"请勿弯折"（`Content-Type: application/json`），快递员就知道这个包裹是什么、怎么处理。`Accept: application/json` 就是客户说"请用中文回复，别用法语"——告诉服务器我能处理的格式。

| Header | 示例 | 作用 |
|--------|------|------|
| `Content-Type` | `application/json` | 告诉对方 Body 是什么格式 |
| `Content-Length` | `348` | Body 的大小（字节），接收方知道读多少数据后停止 |
| `Accept` | `application/json` | 客户端告诉服务器"我能处理这些格式" |
| `Accept-Encoding` | `gzip, br` | 客户端告诉服务器"我支持这些压缩算法" |

#### 认证类

| Header | 说明 |
|--------|------|
| `Authorization: Bearer <token>` | 现代 API 最常用的认证方式（JWT token） |
| `Authorization: Basic <base64>` | `base64(username:password)`，必须走 HTTPS |
| `Cookie` | 浏览器自动带的身份标识 |

#### 缓存类

| Header | 说人话 |
|--------|--------|
| `Cache-Control: max-age=3600` | "这个资源可以缓存 3600 秒" |
| `Cache-Control: no-cache` | "可以缓存，但每次都要先问服务器有没有变" |
| `Cache-Control: no-store` | "别缓存，每次都来我这拿新的" |
| `ETag: "abc123"` | "这个版本的标识是 abc123"，配合 `If-None-Match` 做条件请求 |
| `Last-Modified` | 资源的最后修改时间，配合 `If-Modified-Since` |

> **面试常见问题**：`no-cache` vs `no-store` 的区别？
> `no-cache` 可以缓存，但每次用之前必须向服务器**验证**（发一个条件请求）。
> `no-store` 完全禁止缓存，路由器/CDN/浏览器都不许存。
>
> **生活化类比**：`no-cache` 就像**冰箱里的剩菜**——可以存着，但每次吃之前得先检查有没有坏（验证）。`no-store` 就像**外卖的一次性餐盒**——吃完就扔，不存。`max-age=3600` 就是**食品保质期标签**——在这之前放心吃，过期了就得重新买。

#### CORS 相关

| Header | 说人话 |
|--------|--------|
| `Origin: https://myapp.com` | 浏览器自动带，告诉服务器"请求来自这个域" |
| `Access-Control-Allow-Origin: *` | 服务端说"谁都能跨域调我" |
| `Access-Control-Allow-Origin: https://myapp.com` | 服务端说"只有这个域能调我" |
| `Access-Control-Allow-Methods: GET, POST` | 允许的 HTTP 方法 |
| `Access-Control-Allow-Headers: Content-Type, Authorization` | 允许前端带的自定义头 |
| `Access-Control-Allow-Credentials: true` | 允许带 Cookie 跨域 |

> **CORS 的本质不是"限制前端"，而是"保护浏览器用户"——防止恶意网站用你的登录态去偷数据。** 相关理解参考 [同源策略](https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy)。
>
> **生活化类比**：CORS 就像**写字楼门禁**——你自己公司的员工刷工卡可以随意进出（同源），但隔壁公司的人想来串门，得你公司前台确认"这人可信"并登记后才能进（CORS 白名单）。如果完全没有门禁，任何人都能拿着捡来的工卡混进任何公司。

### 1.6 HTTP 版本演进

| 版本 | 年代 | 核心改进 | 痛点 |
|------|------|---------|------|
| **HTTP/1.0** | 1996 | 诞生，每个请求一个 TCP 连接 | 太慢，每张图都要新握手 |
| **HTTP/1.1** | 1997 | **Keep-Alive**（连接复用），管道化（Pipelining） | 队头阻塞——同一连接上的请求必须按顺序返回 |
| **HTTP/2** | 2015 | **多路复用**（一个连接并行传多个请求），头部压缩（HPACK），服务器推送 | 仍然基于 TCP，TCP 层面的丢包会阻塞所有流 |
| **HTTP/3** | 2022 | 换底层为 **QUIC**（基于 UDP），0-RTT 握手，消除 TCP 队头阻塞 | 部署成本高，需要 UDP 不被防火墙拦截 |

**现成实践：** 绝大多数生产环境已经上 HTTP/2（HTTPS 自动启用）。HTTP/3 正在铺开——取决于你用的 CDN/负载均衡是否支持。

**面试考点：**
- HTTP/2 多路复用解决了什么？HTTP/1.1 的队头阻塞（同一连接上的请求必须串行返回）。
- HTTP/3 为什么快？QUIC 基于 UDP，不用等 TCP 握手；不同流各自独立，一个丢包不影响其他流。

> **生活化类比**：HTTP 版本演进就像**快递行业的升级**——HTTP/1.0 是每寄一个包裹打一次电话叫快递员（每个请求新连接）；HTTP/1.1 是快递员一次上门取走所有包裹（Keep-Alive 复用连接）；HTTP/2 是快递员同时派送多个包裹互不阻塞（多路复用）；HTTP/3 是直接把传统快递公司换成无人机——底层运输方式都变了（TCP → UDP），不受地面堵车影响（TCP 队头阻塞），而且快递员认识你家地址，不用每次重新问路（0-RTT）。

### 1.7 HTTPS 与 TLS

#### 为什么需要 HTTPS

HTTP 是明文传输。你在咖啡店连 WiFi，旁边的任何人抓包都能看到你发的密码。

HTTPS = HTTP + TLS。TLS（Transport Layer Security）在 HTTP 和 TCP 之间加了一层加密，保证：
1. **机密性**：传输内容加密，中间人看不懂
2. **完整性**：内容没被篡改
3. **身份验证**：你确实在和老王聊天，不是有人冒充

#### 加密基础概念（5 分钟速通）

| 类型 | 原理 | 优缺点 |
|------|------|--------|
| **对称加密** | 加解密用同一把钥匙 | 快，但怎么安全地把钥匙交给对方是问题 |
| **非对称加密** | 一把公钥（加密）+ 一把私钥（解密），成对出现 | 公钥可以公开，私钥只有自己知道。慢，比对称慢 100-1000 倍 |
| **混合方案（TLS 实际用的）** | 用非对称加密传输对称密钥，后续通信全用对称加密 | 两者优点结合 |

> **生活化类比**：对称加密就像**两人共用一把钥匙开同一个信箱**——快是快，但怎么安全地把钥匙交给对方是问题（快递寄钥匙？万一被截了）。非对称加密就像**邮政信箱**——任何人都可以往里投信（公钥加密），但只有拿钥匙的人能打开取信（私钥解密）。TLS 的混合方案就是：先用邮政信箱安全地寄一把共享钥匙给对方，之后两人用这把钥匙开普通信箱通信——又快又安全。
>
> **证书链的类比**：你的浏览器不是天生信任一个网站的。信任链就像**护照体系**——你信任一个陌生人的护照，是因为你信任签发护照的政府（根 CA），政府信任各地的出入境管理局（中间 CA），出入境管理局经过核实签发了护照（服务器证书）。自签名证书就像一个**手写纸条**说"我是张三"，没有官方背书——浏览器当然不认，报红屏警告。

**延伸阅读**：[TLS 握手全解析](https://www.cloudflare.com/learning/ssl/what-happens-in-a-tls-handshake/)

### 1.8 动手实践

用 curl 逐行读懂一次完整请求：

```bash
# -v 显示完整过程，包括 TLS 握手和所有 Headers
curl -v https://www.baidu.com

# 只看响应头
curl -I https://www.baidu.com

# 关注输出中的：
# * TLSv1.3 ...        ← TLS 握手信息
# > GET / HTTP/2       ← 请求行（> 开头是发出的）
# < HTTP/2 200         ← 状态行（< 开头是收到的）
# < content-type: ...  ← 响应头
```

Chrome DevTools 实操：
1. 打开 Network 面板
2. 访问一个网站
3. 点第一个请求 → 看 Timing 选项卡
   - **Queueing** → 浏览器排队
   - **Stalled** → 等 TCP 连接
   - **DNS Lookup** → DNS 解析时间
   - **Initial Connection** → TCP 握手 + TLS 握手
   - **Request sent** → 发请求
   - **Waiting (TTFB)** → 等服务器响应（Time To First Byte）
   - **Content Download** → 下载响应体
4. 这就是一次完整请求的时间分解

## 2. TCP/IP 基础

### 2.1 网络分层模型

把网络通信比作寄快递：你在 28 楼扫码寄件，快递员收件、装箱、贴面单、装车、上高速、到目的地后再一层层拆出来送到 28 楼。

每一层只关心自己的工作，不关心其他层的细节——这就是"分层"的设计哲学。

> **生活化类比**：网络分层就像**跨国快递流程**——你写的内容（应用层）被装进信封（传输层），快递公司贴上面单写上地址（网络层），司机装车、飞机运货（链路层）。每一层只做自己的事：飞行员不关心包裹里是合同还是情书，海关不关心飞机走哪条航线。正因如此，你可以把 HTTP 从 1.1 升级到 3.0 而不需要换光纤——应用层和物理层各自独立演进。

#### OSI 七层模型 vs TCP/IP 四层模型

```
┌─────────────────┬──────────────────────────────────┬─────────────────────────┐
│   TCP/IP 四层    │            做什么                 │         类比              │
├─────────────────┼──────────────────────────────────┼─────────────────────────┤
│   应用层         │ HTTP, DNS, WebSocket — 你的代码   │ 写内容、打包            │
│   Application   │ 直接跟这一层打交道                 │                         │
├─────────────────┼──────────────────────────────────┼─────────────────────────┤
│   传输层         │ TCP / UDP — 端到端可靠传输        │ 快递公司选顺丰还是邮政    │
│   Transport     │ 端口号在这一层                     │                         │
├─────────────────┼──────────────────────────────────┼─────────────────────────┤
│   网络层         │ IP — 寻址和路由，找到目的地        │ 地址和路线规划            │
│   Internet      │ IP 地址在这一层                    │                         │
├─────────────────┼──────────────────────────────────┼─────────────────────────┤
│   网络接口层      │ 物理传输（网线、WiFi、光纤）       │ 货车、飞机、高速公路       │
│   Link          │ MAC 地址在这一层                   │                         │
└─────────────────┴──────────────────────────────────┴─────────────────────────┘
```

> **前端工程师不需要背七层模型，但要理解"为什么分层"**：每一层独立演进，HTTP 从 1.1 升到 2 到 3，应用层的代码不关心底层走的是 WiFi 还是光纤。

**数据是怎么包起来的：**

```
用户的 JSON 数据
  → HTTP 加个请求头（应用层）
    → TCP 分段 + 端口号（传输层）
      → IP 加 IP 头和地址（网络层）
        → 网卡加 MAC 地址发出去（链路层）

对方收到后反向一层层拆开。
```

### 2.2 TCP 三次握手与四次挥手

#### 三次握手（建立连接）

```
客户端                                服务器
  |                                      |
  |——— SYN (seq=x) ——————————————————→  |  "你好，我想跟你建立连接"
  |                                      |
  |←— SYN+ACK (seq=y, ack=x+1) ——————  |  "收到了，我也要跟你建立连接"
  |                                      |
  |——— ACK (seq=x+1, ack=y+1) ——————→  |  "好的"
  |                                      |
  |        连接建立，可以传输数据了          |
```

**为什么是三次，不是两次？**
- 如果只有两次，服务器收到 SYN 就认为连接建立了、可以发数据——但可能这是一个早已过期的 SYN 包，客户端早就放弃了，服务器却白白把资源浪费在这个"死连接"上
- 第三次 ACK 让客户端确认：我真的要这个连接；同时也让服务器确认：客户端收到了我的回复，双方都准备好了

> **生活化类比**：三次握手就像**打电话**——
> 1. 你拨号："喂，听得到吗？"（SYN）
> 2. 对方："听到了，你能听到我吗？"（SYN+ACK）
> 3. 你："能听到，我们开始聊吧"（ACK）
>
> 如果只有两次——你说"喂"、对方说"听到了"就以为通话建立了——万一你的"喂"是一小时前对别人喊的、早就挂了、现在正跟另一个人打呢？第三次确认才能保证双方在同一通电话里。

#### 四次挥手（断开连接）

```
客户端                                服务器
  |                                      |
  |——— FIN ——————————————————————————→  |  "我说完了，不发了"
  |                                      |
  |←— ACK ———————————————————————————  |  "收到"（但服务器可能还有数据要发）
  |                                      |
  |←— FIN ———————————————————————————  |  "我也说完了"
  |                                      |
  |——— ACK ——————————————————————————→  |  "收到"
  |                                      |
  |  客户端等 2MSL 后关闭                  |
```

**为什么是四次而不是三次？** TCP 是全双工的（双向车道），每个方向要独立关闭——客户端不发≠服务器不发，所以 FIN 和 ACK 不能合并。

**为什么客户端最后要等 2MSL？** 确保最后一个 ACK 被服务器收到——如果丢了（服务器会重发 FIN），客户端还能重发 ACK。

> **生活化类比**：四次挥手就像**结束一通电话**——
> 1. 你："好的，事情聊完了，我这边没要说的了"（FIN）
> 2. 对方："行，知道了"（ACK）——但对方可能突然想起"哦对了还有一件事..."
> 3. 对方："我也说完了，拜拜"（FIN）
> 4. 你："拜拜"（ACK）
>
> 最后你等 2MSL 再挂电话——这就像你把话筒放下后确实等了两秒才挂断，确保对方听到了你的"拜拜"。如果你秒挂，对方可能对着断线喊"喂？你说什么？"（服务器重发 FIN，但没人 ACK 了）。
>
> 为什么需要四次？因为一边说完不等于另一边也说完了——你是倾诉完想挂，但对方可能还有话。

### 2.3 TCP 与 UDP 的区别

| 维度 | TCP | UDP |
|------|-----|-----|
| **连接** | 有连接（三次握手） | 无连接，直接发 |
| **可靠性** | 保证送达、保证顺序、不丢不重 | 不保证——可能丢、可能乱序 |
| **速度** | 慢（有握手、确认、重传机制） | 快（无额外开销） |
| **适用场景** | HTTP、文件传输、邮件、数据库 | 直播、视频通话、在线游戏、DNS |
| **类比** | 打电话——先接通，确认对方在，一句话一句话说 | 发喇叭广播——只管喊，谁听到算谁的 |

> **生活化类比**：TCP 像**挂号信**——必须签收、保证送达、按顺序投递。丢了？重发。乱了？排好序。UDP 像**群发传单**——发出去就算完成任务，街上风一吹、掉了、撕了、没看见，一概不管。所以直播用 UDP——错过一个画面帧没什么大不了，但要等重传等一秒画面就卡死了。

**为什么 HTTP/3 改用 UDP (QUIC 基于 UDP)？**
TCP 的可靠性保障机制（RTT、重传、保序）反而成了瓶颈——一个数据包的丢失会阻塞后续所有数据（TCP 队头阻塞）。

### 2.4 IP 地址、子网掩码、端口

#### IP 地址

IPv4：`192.168.1.1`，4 个字节（32 位），约 43 亿个——已经分完了。

IPv6：`2001:0db8:85a3:0000:0000:8a2e:0370:7334`，16 个字节（128 位），约 340 万亿个——"地球上每一粒沙子都能分配一个 IP"。

**公网 IP vs 私有 IP：**

| 范围 | 类型 | 说人话 |
|------|------|--------|
| `10.x.x.x` | 私有 | 公司/家庭内网用的 |
| `172.16.x.x ~ 172.31.x.x` | 私有 | 同上 |
| `192.168.x.x` | 私有 | 你家路由器默认发的就是这个 |
| `127.0.0.1` | 回环 | `localhost`，指向自己这台机器 |
| 其他 | 公网 | 互联网上唯一的地址 |

`192.168.1.1` 可以出现在无数家庭的路由器上——它们不在同一个网络空间里。

#### 子网掩码

`192.168.1.0/24` 表示网络号占前 24 位（即 `192.168.1`），后 8 位是主机号。
同一个子网内的设备可以直接通信，不同子网的需要通过路由器转发。

#### 端口

IP 地址定位到"哪台机器"，端口定位到"哪个进程"。

- 范围：0 ~ 65535（16 位）
- `0 ~ 1023`：知名端口（HTTP:80、HTTPS:443、SSH:22、MySQL:3306、PostgreSQL:5432）
- `1024 ~ 49151`：注册端口
- `49152 ~ 65535`：临时端口（客户端发起连接时随机用的）

> `localhost:3000` 的 `3000` 就是端口。前端开发天天见。

### 2.5 NAT 与内网穿透

#### 场景

你家里有 3 台设备连同一个路由器，运营商只给一个公网 IP。
问题：3 台设备共用一个 IP，外网请求进来时路由器怎么知道该转发给哪台？

#### NAT (Network Address Translation)

路由器维护一张**转换表**，把内网 IP+端口 映射为 外网 IP+端口：

| 方向 | 地址 |
|------|------|
| 你的笔记本发出时 | `192.168.1.5:54321 → 120.5.6.7:54321` |
| 服务器回复时 | `120.5.6.7:54321 → 192.168.1.5:54321` |

**NAT 对开发的影响：**
- 你本地 `localhost:3000` 起服务，外网的同事访问不到——因为你的 `192.168.1.5` 是私有 IP
- 解决方案：内网穿透工具（ngrok、frp、Cloudflare Tunnel）在公网上建一个中间服务器做转发

> **生活化类比**：NAT 就像**公司总机转分机**——外面的人只能打公司总机号码（公网 IP），总机前台根据分机号（端口）帮你转接到对应工位（内网设备）。你同事坐在隔壁工位只能拨分机号直接联系你（内网通信），但外人没总机号码根本联系不到你。内网穿透就是**你给了客户一个虚拟号码，客户打这个号会自动转接到你的分机**。

## 3. DNS 与域名

### 3.1 DNS 是什么

**DNS (Domain Name System)** 把人类可读的域名（`baidu.com`）翻译成机器可读的 IP 地址（`110.242.68.66`）。

它是互联网的电话本。没有 DNS，你得记住每个网站的 IP 地址。

> **生活化类比**：DNS 就是互联网的**114 查号台**——你说"帮我查一下百度"，114 回复"百度的电话是 110.242.68.66"。你不需要背电话号码簿，只要知道名字就能查。

### 3.2 DNS 解析全流程

当你在浏览器输入 `www.example.com` 并按回车：

```
1. 浏览器自己的 DNS 缓存
   └─ 命中了？返回 IP 地址

2. 操作系统缓存 (hosts 文件)
   └─ 命中了？返回 IP 地址

3. 本地 DNS 解析器 (Local DNS Server)
   通常是你家路由器或运营商提供的 DNS 服务器（如 114.114.114.114）
   └─ 命中了？返回 IP 地址

4. 根域名服务器 (Root DNS)
   问："www.example.com 在哪？"
   答："我不知道，但我知道 .com 的服务器在哪"

5. 顶级域名服务器 (TLD DNS - .com)
   问："example.com 在哪？"
   答："去问 example.com 的权威 DNS 服务器"

6. 权威 DNS 服务器 (Authoritative DNS - example.com)
   问："www.example.com 的 IP 是什么？"
   答："1.2.3.4"

7. Local DNS Server 把结果缓存起来，下次直接返回
```

**两种查询模式：**

| 类型 | 流程 | 类比 |
|------|------|------|
| **递归查询** | 你问 A → A 问 B → B 问 C → 结果原路返回 | 你让助理去查，助理帮你跑完全部 |
| **迭代查询** | 你问 A，A 说"去问 B"，你问 B，B 说"去问 C"，自己一步步查 | DNS 服务器之间用这种 |

> **生活化类比**：递归查询就像**你让秘书去订机票**——秘书打了一圈电话（航空公司、票务、酒店），最后只告诉你"订好了"。迭代查询就像**你亲自跑腿**——去民航局问"飞机在哪"，对方说"去问机场"，你去机场问，对方说"去问登机口"——你一步步跑，但每一步都知道下一步找谁。

> 实际过程中：客户端 → Local DNS 走递归，Local DNS → 根/TLD/权威走迭代。

**用 dig 自己追踪一次解析：**

```bash
# 看完整解析过程（+trace 走迭代查询）
dig +trace www.example.com

# 输出会展示从根 → .com → example.com 权威的每一步
```

### 3.3 DNS 记录类型

| 记录类型 | 含义 | 示例 | 说人话 |
|---------|------|------|--------|
| **A** | 域名 → IPv4 地址 | `example.com → 1.2.3.4` | "这个域名对应这个 IP" |
| **AAAA** | 域名 → IPv6 地址 | `example.com → 2001:db8::1` | 同上，IPv6 版 |
| **CNAME** | 域名 → 另一个域名 | `www.example.com → example.com` | "www 的地址跟着主域名走" |
| **MX** | 邮件服务器 | `example.com → mail.example.com` | "发给 @example.com 的邮件去哪" |
| **NS** | 权威 DNS 服务器 | `example.com → ns1.dns.com` | "这个域名的 DNS 记录归谁管" |
| **TXT** | 任意文本 | `v=spf1 ...` | 做域名验证、SPF 邮件策略 |

**A 记录 vs CNAME 什么时候用：**

- 你的服务器有固定 IP → 用 **A 记录**
- 你想把 `www.domain.com` 指向 `domain.com`（跟着变、不用分别配 IP）→ 用 **CNAME**
- CNAME 不能用于根域名（`example.com`），那是协议规定——解析根域名必须用 A 记录

> **生活化类比**：A 记录就像**家庭地址**——"我家住在北京市朝阳区xxx路 1 号"，直接告诉你门牌号。CNAME 就像**转寄地址**——"我不常在家，你寄到公司地址，公司会转给我"，这样你搬家了（IP 变了）只要更新公司地址（主域名）就行，不用通知所有寄件人。但根域名不能用 CNAME 就像一个**没有上级引路人的人也不能靠转寄**——你总不能说"我住的地方就是跟着我住的地方走"。

**实战：买完域名后你需要配什么：**

```
1. A 记录: example.com → 你的服务器 IP (1.2.3.4)
2. CNAME: www.example.com → example.com
3. （可选）MX 记录 → 指向邮件服务商
4. （可选）TXT 记录 → 用 ACME 验证域名所有权配 HTTPS
```

### 3.4 从买域名到网站可访问

这是一个完整链路，建议实际走一遍：

```
1. 买域名
   → 阿里云万网 / 腾讯云 DNSPod / Namecheap / Cloudflare

2. 买 / 准备服务器
   → 云服务器 ECS / 轻量应用服务器 / VPS
   → 得到一个公网 IP（如 1.2.3.4）

3. 在服务器上部署你的应用
   → 前端静态文件 + 后端 API（Nginx 反代）
   → 确保访问 http://1.2.3.4 或者 http://1.2.3.4:3000 能通

4. 在域名服务商配置 DNS
   → 添加 A 记录: yourdomain.com → 1.2.3.4
   → 等待 DNS 传播（通常几分钟到几十分钟）

5. 配 HTTPS
   → 用 acme.sh / certbot 申请 Let's Encrypt 免费证书
   → 配 Nginx 使用证书，HTTP 重定向到 HTTPS

6. Done! https://yourdomain.com 可访问
```

### 3.5 CDN 原理

**CDN (Content Delivery Network)** 让你访问北京的网站时数据从北京的服务器来，而不是从美国横跨太平洋。

#### 工作原理

```
1. 你的域名（如 cdn.example.com）CNAME 指向 CDN 厂商的域名

2. 用户请求 cdn.example.com
   → DNS 解析到 CDN 厂商的 DNS
   → CDN DNS 根据请求来源 IP 判断用户位置
   → 返回最近的边缘节点 IP

3. 如果边缘节点有缓存 → 直接返回
   如果没有 → 回源到你配置的源站拉取，缓存后返回
```

#### 好处
- 用户访问快（就近响应）
- 源站压力小（大部分请求被 CDN 节点拦截）
- 抗 DDoS（多节点分散攻击流量）

> **生活化类比**：CDN 就像**连锁便利店**——你要买一瓶可乐，不用跑到可乐工厂（源站），楼下便利店（边缘节点）就有。而且便利店遍布全城，你走到哪儿都能就近买到。万一有人恶意扫空一家店（DDoS），其他分店的货还在，不影响别的顾客。
